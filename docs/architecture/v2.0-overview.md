# Architecture: SMS Monitor v2.0 — Overview

**Status:** ✅ All sprints complete (Sprint 0-9)
**Last Updated:** 2026-06-18
**Owner:** jokos

---

## What is v2.0?

v1.0 is a single-tenant Android app that captures SMS messages and POSTs them to a test Express server. It works. It shipped. It is sealed in [`src-backup/v1.0/`](../../../src-backup/v1.0/).

**v2.0 is the multi-tenant SaaS-ification of v1.0.** Tenants (think: a school, a payroll company, a CRM vendor) sign up, claim Android devices, configure keyword filters, and receive matched SMS at their own backend URL.

## Data Model (shared schema + `tenantId` on every row)

```
Tenant ──┬── Users (OWNER/ADMIN/VIEWER)
         ├── Devices (HMAC-signed)
         ├── ClaimCodes (one-time onboarding)
         ├── Keywords (per-tenant or per-device)
         └── SmsLogs (one row per matched SMS, dedup by deviceId+smsId)
```

**Isolation strategy:** every domain table has `tenantId`. A Prisma client extension reads the active tenant from `AsyncLocalStorage` (set by JWT-guard middleware) and auto-injects `where: { tenantId }` on every query. A test suite asserts that no raw query path can bypass it.

Full schema lives in [`backend/prisma/schema.prisma`](../../../backend/prisma/schema.prisma). Zod schemas for the wire format live in [`packages/shared-types/`](../../../packages/shared-types/).

## Architecture Decision Records (ADRs)

Decisions made during the v2.0 planning discussion. Each is locked in; reversing any of them is a sprint-level undertaking.

### ADR-001: Multi-tenancy = shared schema + `tenantId`

- **Date:** 2026-06-08
- **Status:** Accepted
- **Context:** Need to support dozens-to-hundreds of tenants cheaply and operationally simply.
- **Options considered:** (A) shared schema + `tenantId`, (B) schema-per-tenant, (C) database-per-tenant
- **Decision:** Option A. One Postgres, every domain table has `tenantId`, Prisma middleware enforces.
- **Rationale:** Cheapest to host (one DB), easiest to back up, easiest to analyze. Mitigates cross-tenant leak risk via middleware + tests.
- **Consequences:** Any query that forgets `tenantId` could leak data. Mitigated by Prisma extension + a regression test suite that scans for raw queries.

### ADR-002: Backend = Node.js + NestJS + Prisma

- **Date:** 2026-06-08
- **Status:** Accepted
- **Context:** Reuse existing Node skill; need opinionated architecture for multi-tenant + auth.
- **Options considered:** (A) Node/NestJS, (B) Python/FastAPI, (C) Go/Fiber, (D) Java/Spring
- **Decision:** Option A. NestJS gives modules + DI + guards + interceptors out of the box; Prisma gives compile-time types matching the mobile app's domain model.
- **Consequences:** Node's single-threaded CPU model. Acceptable for our workload (one POST per matched SMS, low fan-out).

### ADR-003: Web = Next.js 14 (App Router) + shadcn/ui

- **Date:** 2026-06-08
- **Status:** Accepted
- **Context:** Tenant admin dashboard, responsive, will eventually be white-label-embeddable.
- **Options considered:** (A) Next.js + shadcn/ui, (B) Vite + React SPA, (C) SvelteKit, (D) defer to v2.x
- **Decision:** Option A. Best ecosystem for admin dashboards, server components fit multi-tenant data flow, Tailwind + shadcn/ui gives the "responsive on phone" out of the box.
- **Consequences:** App Router still has rough edges. Pin to Next 14.2 to avoid `experimental_*` flags.

### ADR-004: Device onboarding = QR code (primary) + manual code (fallback)

- **Date:** 2026-06-08
- **Status:** Accepted
- **Context:** Tenant admins need to onboard physical Android devices with minimal friction.
- **Options considered:** (A) QR, (B) manual 6-digit, (C) deep link
- **Decision:** A + B. QR by default; manual fallback for cameras-broken or admin-on-phones.
- **Consequences:** Two UI surfaces to maintain. Both share the same `ClaimCode` server-side resource.

### ADR-005: v1.0 → v2.0 = hard break on the Android app

- **Date:** 2026-06-08
- **Status:** Accepted
- **Context:** v1.0 app has no concept of tenant; v2.0 does. Migrating in-place adds a compat layer we'd maintain forever.
- **Options considered:** (A) hard break (fresh install), (B) in-place migration reusing v1.0 UUID
- **Decision:** Option A. Documented in `CHANGELOG`; the web admin onboarding walks users through the QR claim.
- **Consequences:** v1.0 users must re-install. Mitigated by a clear migration guide.

### ADR-006: Monorepo with npm workspaces

- **Date:** 2026-06-08
- **Status:** Accepted
- **Context:** Shared TypeScript types between backend and web.
- **Options considered:** (A) monorepo, (B) separate repos
- **Decision:** Option A. `packages/shared-types/` exports Zod schemas consumed by both backend (for request validation) and web (for form validation).
- **Consequences:** Adds a tool (npm/pnpm workspaces). Acceptable — standard for the Node + TS ecosystem.

### ADR-007: Build backend + web first; keep `app/` at v1.0

- **Date:** 2026-06-08
- **Status:** Accepted
- **Context:** The Android app is shipping and working. The backend is the actual production control plane.
- **Options considered:** (A) backend + web first, app stays v1.0; (B) all three at once; (C) vertical slice (one tenant, one device, all three); (D) web first
- **Decision:** Option A. Faster path to a production system; the v1.0 app can be repointed at the new backend with zero code changes.
- **Consequences:** The `app/` won't have multi-tenant features until Sprint 3. Acceptable for the SaaS pilot.

## Cross-cutting concerns

### Authentication
- **Users** → JWT in httpOnly cookie. RS256, rotating keys (HS256 for now, RS256 in Sprint 2).
- **Devices** → `Authorization: Bearer <api_key>` + `X-Device-Signature: <hmac>` + `X-Device-Timestamp`. Replay protection via 5-minute window.
- **Tenants** → JWT carries `tenantId`; `AsyncLocalStorage` makes it implicit in every request.

### Rate limiting
- NestJS `@nestjs/throttler` on `/auth/*`, `/devices/claim`, `/sms/ingest`. Default 60 req/min per IP.

### Background jobs
- BullMQ on Redis. Use cases: SMS forwarding retries, claim-code expiry sweeps, retention pruning.

### Observability
- Sprint 4: Sentry (errors) + Axiom (logs). For now, structured JSON logs via NestJS `Logger`.

### Data retention
- Per-tenant `retentionDays` (default 90). `pruneOldLogs()` job runs nightly.

## Versioning & migration

- `v1.0` → `v2.0` is a hard break (see ADR-005). The v1.0 codebase is sealed in [`src-backup/v1.0/`](../../../src-backup/v1.0/).
- `v2.0.x` (post-Sprint 5) follows SemVer. No more hard breaks planned; additive only.
